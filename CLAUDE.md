# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

INFINITE CARTRIDGE (Axiometa Ă— Anthropic Jul 2026 hackathon): a self-refilling game console. A Python pipeline has Claude design and implement a tiny Arduino game, gates it through a static contract check and a real `arduino-cli` compile (with LLM repair on failure), flashes it to an Axiometa Genesis Mini (ESP32-S3) handheld. `--loop` mode is buffered: after the opening game is flashed, a **pipelined producer** forges ahead until 10 unplayed games are banked (`library/_queue.json`; a restart resumes it, and loop shutdown trims it to one warm opener â€” the rest re-forge fresh next session); each game-over pops the oldest and flashes its **prebuilt binaries** (`library/NNN_*/build/`, exported at forge time via `compile --output-dir`, flashed via `upload --input-dir` in ~10 s, no recompile, no race with the producer's compiler). Each game also gets an LLM-composed AY chiptune theme that starts on a **kolobox** speaker box (real ZX Spectrum AY chips over HTTP) the moment the game is flashed.

## Commands

```bash
# One-time setup
pip install -r requirements.txt        # anthropic, pyserial, python-dotenv, spectrumizer (brings mido)
arduino-cli core update-index && arduino-cli core install esp32:esp32
arduino-cli lib install "Adafruit GFX Library" "Adafruit ST7735 and ST7789 Library" "Adafruit NeoPixel"
cp example.env .env                    # then set ANTHROPIC_API_KEY

# Run (all via main.py)
python main.py --theme "crabs in space"    # forge one game and flash it
python main.py --loop                      # forge a fresh game on every game-over
python main.py --pregen 5                  # bank 5 games into library/ without flashing
python main.py --flash library/<NNN_name>  # reflash a banked game
python main.py --list                      # catalogue of banked games
python gui.py                              # web GUI over the same verbs (localhost; library browser only under /dev)

# Firmware only (no LLM) â€” compile/flash whatever is currently in forge_sketch/
arduino-cli compile --fqbn "$FORGE_FQBN" forge_sketch
arduino-cli upload -p "$FORGE_PORT" --fqbn "$FORGE_FQBN" forge_sketch

# Regenerate the sprite atlas + prompt catalog (after editing CATALOG in spritegen.py)
python spritegen.py [--preview out.png]

# Labeled per-genre cast review PNGs (settings by roll weight, pools on real GAME_BG)
python sprite_sheets.py                # writes sprite_sheets/genre_<name>.png

# Re-render a cartridge's theme.pt3 + theme.wav from its saved theme.json (no LLM)
python -m music.score library/<NNN_name>
```

Configuration is env vars read in `config.py` (`.env` supported): `FORGE_FQBN` (default `esp32:esp32:axiometa_genesis_mini`; with the vanilla `esp32:esp32:esp32s3` core the `P1_IO0`-style port macros in `engine.h` must be replaced with raw GPIO numbers), `FORGE_PORT` (default `/dev/cu.usbmodem1101`; `device.port()` falls back to globbing usbmodem/ttyACM/ttyUSB), `FORGE_MODEL`, `ZX_SPECTRUM_CHIP_URL` (the music box's `http://<host>:8080`; unset or empty disables music entirely).

There is no test suite or linter. Validation of generated games is `game_factory.static_check()` plus the real compile; expect first compile to be slow (cold cache), later ones ~30â€“60 s.

## Architecture

Two halves â€” a Python host pipeline and ESP32 firmware â€” joined by a prompt contract and a serial JSON protocol.

**Host pipeline (Python).** `main.py` orchestrates: ontology roll â†’ skin (fiction **and an ASCII level map**) â†’ design check â†’ implement â†’ static check â†’ compile â†’ repair (â‰¤3 attempts, compiler errors fed back into the repair prompt) â†’ save to `library/` â†’ flash (+ start music); the theme music composes on a parallel thread under implement+compile. `forge_one` is split into `llm_stage` (roll/skin/implement â€” API only) and `build_stage` (static check/compile/repair/save â€” sole owner of `forge_sketch/` and `BUILD_TMP`); the loop producer and `--pregen` run them as `pipeline()`'s two workers so game N+1 is authored while game N compiles (~2Ă— buffer fill). The LLM worker rolls against `history() + pending` â€” coords of authored-but-unsaved games â€” so anti-repeat holds across in-flight games; single-shot forges stay sequential.
- `ontology.py` â€” **Python rolls the dice, the LLM only decorates.** 12 genre archetypes (catcher, dodger, racer, shooter, defender, sorter, stacker, platformer, breaker, crosser, snake, climber), each with compatible objectives/threats/twists, numeric param ranges, and a prose spec-sheet template. `roll(history)` picks coordinates with anti-repeat (genre âˆ‰ last 4, genre/objective pair âˆ‰ last 8, twist/setting âˆ‰ last 3), applies pace multipliers, and caps count-goals by the spawn economy so games are **winnable by construction**. The filled spec-sheet rides into the implement prompt.
- `game_factory.py` â€” the Claude calls (`skin`, `implement`, `repair`) plus two non-LLM gates: `design_check` (title/blurb/hint lengths, sprite ids validated against `sprites.h`) and `static_check` (banned patterns â€” `delay`, `millis`, `while`, `#include`, `String`, `tft`, `matrix.`, heap allocation, matched after stripping comments/strings â€” required symbols incl. `GAME_GOAL`/`GAME_VERB`/`GAME_HINT`/`eProgress`, â‰¤260 lines; runs on code with the map block already appended). `skin()` must not alter rolled coordinates; the merge overwrites any it touched.
- `device.py` â€” arduino-cli wrapper (`compile_sketch`, `upload` with retries for port re-enumeration) and `wait_for(ev)`, which blocks reading serial JSON events (this is all `--loop` is).
- `config.py` â€” paths + env, and it assembles the LLM inputs: `CONTRACT` is ENGINE_PROMPT.md verbatim; `GAME_EXAMPLES` is a sorted glob of root-level `example*.ino`.
- `library/*/meta.json` carries the rolled coordinates â€” it is the anti-repeat memory that `history()` feeds back into `roll()`.
- **Wildcards & mercy**: `WILD_P` (25%) of rolls go wild â€” guest sprites from another setting, avatar-is-a-foe swap, two-setting blend, or a `MUTATORS` entry with a mandatory mechanical spec line; never two wild games in a row (`wild_kind` in meta). Difficulty is centrally merciful: `roll()` softens hazard speeds/ramps after pace (never player params), clamps frantic+collision stacks, and the spec suffix mandates `GAME_LIVES 10`, a +50% ramp cap and an 8-second ease-in; the engine adds a 1 s `eLoseLife` cooldown (no multi-hit drains) and shows 10+ lives as matrix pips. Skin runs at temperature 1.0, implement/repair at 0.6.

**Firmware (`forge_sketch/`).** `engine.h` is the entire console: ST7735 160Ă—80 with a GFXcanvas16 framebuffer, PICO-8 palette `PAL[16]`, d-pad on an ADC resistor ladder (orientation in the `PADMAP` table), quadrature encoder via ISR, 5Ă—5 NeoPixel matrix as the lives HUD, and a 30 fps TITLEâ†’PLAYâ†’OVER shell that owns timing, input polling, score strip, and blitting. `forge_sketch.ino` just calls `engineSetup()/engineLoop()`. `forge_sketch/game.ino` is the cartridge slot â€” **machine-overwritten on every forge/flash**, never hand-edit it expecting persistence; permanent copies live in `library/`.

The Arduino build concatenates all `.ino` files in `forge_sketch/`, so `game.ino` sees `cv`, `inp`, `PAL`, `eScore()` etc. from `engine.h` with no include â€” which is why `#include` is on the banned list.

**The contract.** `ENGINE_PROMPT.md` is simultaneously the human documentation of the game API and, verbatim, the LLM system prompt. `example.ino` (FOREST VIGIL, tagged "complete correct game") and `example2.ino` (PITLANE DASH, tagged "target depth â€” match this") â€” real forged cartridges with their forge-appended map blocks stripped â€” are appended as few-shot examples, with prompt caching on the whole prefix; dropping another `example*.ino` at the root silently changes generation. When the engine API changes, four places must stay in sync: `engine.h` (what exists), `ENGINE_PROMPT.md` (what games are told), `BANNED`/`REQUIRED` in `game_factory.py` (what is enforced), and the genre spec-sheets in `ontology.py` (what each genre is told to call). Games declare `GAME_GOAL`/`GAME_VERB`/`GAME_HINT`/`GAME_BG` and call `eProgress(d)`; the engine draws the goal HUD and fires the win itself â€” cartridges predating this contract no longer compile (re-forge the library with `--pregen`).

**Level maps.** The skin stage authors a 20Ă—8 ASCII map (`.#PGS` + legend letters â†’ prop sprites; `MAPS` in `ontology.py` holds per-genre briefs for the author and usage rules for the implementer). `design_check` validates shape/legend/anchors/density; `game_factory.map_code()` emits the `GAME_MAP*` constants which `forge_one` **appends to the generated code** â€” the implement LLM never writes them (and the exemplars deliberately omit them; verification appends a fixture block before compiling). The engine fills `PAL[GAME_BG]`, renders the map each frame under the game layer, and exposes `mapSolid(x,y)` / `mapCount(ch)` / `mapX/mapY(ch,i)`; games are forbidden from drawing their own static scenery or calling `fillScreen`.

**Sprites.** `spritegen.py` bakes ~360 curated, hand-named sprites from the CC0 packs in `assets/` (micro-roguelike, pico-8-platformer, Paper-Pixels, modern-city, racing) into `forge_sketch/sprites.h` and rewrites the catalog block between `<!-- SPRITES:BEGIN/END -->` markers in `ENGINE_PROMPT.md` â€” never edit either generated block by hand; edit `CATALOG` (topic-grouped entries), `CMAP` (hand color tables for MR/P8) or `OVERRIDE` (fixups over nearest-PICO-8 for the auto-quantized packs) in `spritegen.py` and rerun. Two banks share one id space: 8Ă—8 texels drawn 2Ă— (ids < `SPR16_BASE`) and native 16Ă—16 drawn 1Ă— (vehicles, city props â€” same 16Ă—16 screen footprint, 4Ă— detail); racing PNGs are one-file-per-sprite, NEAREST-downscaled (BOX averaging would break exact-color overrides) and rotated to face right. The engine exposes `spr(id, x, y, flags)` and `sprT(id, x, y, palColor, flags)` (tinted silhouette, used for damage flashes) with `SF_FLIPX`/`SF_FLIPY`, `SF_BIG` (32Ă—32), `SF_SMALL` (8Ă—8). Sprite names are the LLM's vocabulary: if a sprite isn't in the ENGINE_PROMPT.md catalog, generated games can't use it, and a name that misdescribes its pixels will produce thematically wrong games. Casting is contextual: `SETTINGS` kits in `ontology.py` decide per setting which sprites serve as avatars/foes/goods/props (with per-setting avatar overrides â€” speedway/city cast vehicles and gunners), and `SETTING_WEIGHTS` biases genres toward matching settings; `python ontology.py` asserts every kit reference exists in the regenerated atlas.

**Theme music.** `compose_theme_start()` fires as implement begins and runs the whole music pipeline on a daemon thread under implement+compile (it needs nothing from the build): `game_factory.compose()` has the LLM write a strict-JSON chip-music score (pitch names on a sixteenth grid, lead/bass/harmony + `K/S/H/O` drums, tempo, and "flavor" arrangement flags), gated by `music.score.check()` with one repair retry â€” the crucial rule is register separation (lead strictly highest, bass strictly lowest) because spectrumizer's arranger assigns AY voices by pitch, not by track. `music/score.py` bakes `theme.json/mid/pt3/wav` into a `library/.theme_*` scratch dir (startup janitor sweeps orphans): mido writes the MIDI, `music/midi2pt3.py` (wrapping spectrumizer, stamping the papaya tone-table byte) arranges the `.pt3`, and spectrumizer's software AY renders the `.wav` snippet (â‰¤30 s) that the GUI plays (`/api/wav/<dir>`, â–¶ card in the console on `@music`, â™ª on `/dev` library cards). After save, `compose_theme_finish()` adopts the files into the cartridge dir and **pre-loads** the `.pt3` onto kolobox (`/songs?name=<dir>.pt3`, no autoplay; names must start alphanumeric â€” `NNN_slug` qualifies). Only when the flash fully completes does `play_for(dir)` start it â€” a tiny `POST /play?song=<dir>.pt3` (save+play fallback if the box missed the upload), or `POST /stop` for theme-less legacy cartridges. From then on the soundtrack follows the shell via `on_console_event`, run by whoever owns the port: `over` silences the box, `play` restarts the theme (replays re-sync from the top). `library/.nowplaying` names the live cartridge; a CLI `--theme` forge lingers watching after LIVE (Ctrl-C exits) while a GUI forge exits and hands the lifecycle to gui.py's `console_pump`, which stays off the wire whenever a `wait_for` heartbeats `library/.port`. Music is never fatal: compose/convert failures ship the game silent, the kolobox client never raises, and `--pregen` composes without playing. `music/spectrumizer-main/` is vendored upstream â€” don't modify it.

**Serial protocol.** The engine emits one-line JSON events at 115200 baud â€” `{"ev":"boot"|"blank"|"title"|"play"|"life"|"over"|"pong"}`, with `over` carrying win/score/title and `pong` carrying `shell` (`title|brief|play|over|blank`) â€” and accepts `start`, `title`, `bless`, `ping`, `frame` commands. **Boot gate**: the console powers up as an empty INSERT CARTRIDGE slot (re-announcing `blank` every 2 s, buttons dead) until a host sends `bless` â€” but blessing is reserved for active runs: `device.bless()` fires after each flash, and `wait_for` (loop/forge linger) revives a mid-run replug, while gui.py's idle `console_pump` deliberately never blesses (it silences the kolobox on blank instead, once per episode). A plugged-in console with the GUI merely open stays at INSERT CARTRIDGE until Forge/Loop/Flash is pressed; a power cycle away from the forge likewise leaves the console blank (the cartridge "evaporates on powerdown"; flash is intact, revival is instant on reattach). `frame` replies `@F <base64 of the 160Ă—80 RGB565 framebuffer>` (25,600 B; native USB CDC makes this ~50â€“80 ms despite the nominal baud). Event names are load-bearing: `main.py`'s loop mode literally waits for `"over"`, and `device.state()` pings before a delayed flash so the loop never flashes mid-game (only on the game-over screen, or at startup). Two hostage-proofings in `wait_for`: with `on_blank="return"` (loop only) a `blank` arriving *after* any sign of life means the console rebooted out of a live game (crashed cartridge, replug) and is handed back to the caller as a game-over so the loop rotates the cartridge instead of deadlocking (a blank before any other event is still blessed â€” that's a boot announce whose `bless()` was missed); and after a `SerialException` reconnect it probes `state()` â€” an awaited event that fired into the dead port is recovered from the shell still showing it. **TFT mirror**: the GUI's `/api/screen` touches `library/.mirror` ("someone is watching"); whichever process owns the port â€” the loop child inside `wait_for(mirror=True)`, else gui.py's `console_pump` â€” then requests frames every ~0.6 s into `library/.screen`, which `/api/screen` serves raw to the page's canvas. Frame grabs briefly stall the 30 fps game loop, so nothing is requested when no one watches.

**`library/`** (gitignored) â€” banked cartridges as `NNN_slug/` dirs holding `game.ino`, `design.json`, `meta.json` (music entries feed the compose anti-repeat), and `theme.{json,mid,pt3,wav}`. Failed forges leave an autopsy in `library/_failed_HHMMSS/` (code + errors) and the device keeps the previous game. `library/forge.py` is the untracked legacy single-file version of the whole pipeline, superseded by the `main`/`game_factory`/`device`/`config` split â€” don't extend it.

## Hardware calibration

All knobs are in `engine.h`: swap `PADMAP` rows if d-pad directions feel wrong; ADC band thresholds in `padBand()`; `ENC_A/ENC_B/ENC_SW` defines (`ENC_SW` stays commented out until A/B are verified); the panel is **true-polarity â€” keep `invertDisplay(false)`** (INVON silently negated every frame for the first ~87 carts; when in doubt compare the panel against the GUI mirror, which shows the framebuffer before panel polarity); banked `build/` bins exported before 19 Jul 2026 ~13:00 carry the inverted engine (`--flash` recompiles so it's harmless, and the then-queued 085/086 prebuilts were rebuilt); try `INITR_MINI160x80_PLUGIN` in `engineSetup()` if the image is offset; SPI runs at 26 MHz (40 MHz if stable).
